## Fixes Applied

### Fix 1: Block workout generation failures (BLOCK-02)

**Failure:** "Couldn't generate workouts" — 502 error after ~79s.

**Root cause:** `max_tokens: 8192` too small for multi-week blocks. Claude response truncated mid-JSON, causing `stop_reason: 'max_tokens'` and empty tool_use blocks.

**Fix:**
- Increased `max_tokens` from 8192 to 16384 in `claude-client.ts`
- Added truncation detection: check `stop_reason === 'max_tokens'` before parsing
- Added diagnostic logging when no tool_use block found
- Updated loading message to set user expectations ("1-2 minutes")

### Fix 2: Multi-sport day validation rejection (BLOCK-02)

**Failure:** Validation rejected legitimate multi-sport days (bike+run on same date).

**Root cause:** Duplicate detection used `w${weekNumber}-${date}` as key, treating any two workouts on the same date as duplicates.

**Fix:**
- Changed key to `${date}-${sport}` — only duplicate same-sport workouts on same date are rejected
- Updated `external_id` format to include sport suffix
- Added test for multi-sport day acceptance

### Fix 3: Day preferences UI removal

**Failure:** BLOCK-15 — AI consistently ignored day preferences despite multiple prompt engineering attempts.

**Root cause:** Claude Sonnet 4.6 cannot reliably honor day-of-week scheduling constraints while also generating structured workout content.

**Fix:**
- Removed all day preferences UI from block-setup.tsx (AddPreferenceSheet, DayPreferenceRow, sport requirements)
- Deleted day-preferences.test.tsx
- Documented learnings in plans/future-improvements.md (split scheduling from content generation)
- Removed 6 test cases from CSV

### Fix 4: Locked block not shown on plan tab / dashboard (BLOCK-06)

**Failure:** After lock-in, plan tab showed "Start Block Setup" and dashboard showed "Plan your first block".

**Root cause:** `getActiveBlock()` only queried for `status = 'in_progress'`, but `lockBlock()` sets status to `'locked'`. Locked blocks were invisible.

**Fix:**
- Refactored `getActiveBlock()` to use two sequential queries: prefer `in_progress`, fall back to `locked`. This gives deterministic precedence if both statuses coexist.
- Updated tests to verify the two-query fallback behavior

### Fix 5: Double navigation on lock-in (BLOCK-06)

**Failure:** Plan screen appeared to push twice after lock-in.

**Root cause:** `handleLockIn` called `router.replace('/(tabs)/plan')` on success, AND the `step === 'done'` screen rendered with a "Go to Plan" button doing the same navigation.

**Fix:**
- Removed automatic navigation from `handleLockIn` — let the success screen be the single navigation path

## Learnings

- Claude Sonnet 4.6 cannot reliably honor day-of-week scheduling constraints while generating structured content. Future approach: deterministic scheduling in code, AI for content only.
- `max_tokens` must be sized for worst-case block length (16384 for multi-week blocks)
- Always check `stop_reason` before parsing Claude API responses
- Block state machine: `locked` is an active state from the user's perspective — queries for "active" blocks must include it
- Season setup flow doesn't create goal rows — race data is lost after skeleton generation (logged as future improvement)
